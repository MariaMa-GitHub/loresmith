# Week 3 — First end-to-end answer

*2026-04-16*

## What shipped

1. **Dense retrieval** — pgvector cosine ANN search with scores normalized as `1 - distance`.
2. **RRF hybrid fusion** — Merged BM25 and dense rankings using reciprocal rank fusion (k=60), with a source map ensuring BM25-only passages still carry citation URLs.
3. **RAG pipeline** — Full `RAGPipeline` class: embed query, retrieve via BM25+dense, fuse, prompt via Jinja, stream Gemini answers. Both retrieve and generate operations traced in Langfuse.
4. **Jinja prompt template** — `answer.j2` with inline citation instructions and prompt-injection warnings.
5. **Langfuse v4 tracing** — Migrated to the v4 SDK's `start_as_current_observation` API with graceful noop fallback when credentials are absent.
6. **FastAPI streaming endpoints** — `/chat` (SSE) and `/games` (metadata), with lazy per-game BM25 index rebuilds and per-game pipeline caching in `app.state`.
7. **Minimal Next.js UI** — Game picker landing page, streaming chat interface consuming SSE in real time, `MessageBubble`, `GamePicker`, and `ChatView` components.
8. **Eval dataset expanded** — Added 40 new questions (hades-061 to hades-100) across factual, multi-hop, ambiguous, and adversarial strata for a total of 100 questions.

## What I learned

The trickiest part this week wasn't the architecture — it was the embedding API semantics. pgvector's `<=>` operator returns *distance*, not similarity, so I had to reverse it (`1 - distance`) for ranking. More importantly, Langfuse's v4 SDK is a breaking change wrapped in a minor bump: the old `client.trace(name=...)` API vanished entirely in favor of `start_as_current_observation`. I had to rewrite the entire tracer wrapper, but the new pattern is cleaner — manual span management instead of context-manager magic.

The hybrid fusion design taught me something about data plumbing too. BM25 hits don't carry source URLs, but dense hits do (via the passage join). When RRF picks a BM25-only result, there's no citation. Building a separate `{passage_id: source_url}` map and threading it through the fusion function felt inelegant at first, but it solved the real problem: dense passages win on citation, and BM25-only passages still get retrieved.
