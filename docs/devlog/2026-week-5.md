# Week 5 — Reranker, semantic cache, verifier, typed tools, and ablation harness

*2026-04-24*

## What shipped

1. **Alembic migration 007** — Added `corpus_revision` and `embedding_identity` columns to the semantic cache table so cache entries are scoped to the current ingested corpus and embedding model. Old rows become unreachable after a re-ingest without a schema change.

2. **Cross-encoder reranker** — `CrossEncoderReranker` wraps `sentence-transformers` `CrossEncoder` (`ms-marco-MiniLM-L-12-v2`) to re-score the top candidates from BM25+dense retrieval before generation. `NullReranker` is a drop-in when reranking is disabled.

3. **Semantic cache** — `SemanticCache` stores `(question_embedding, answer, citations)` rows keyed by `(game_slug, corpus_revision)`. A similarity threshold check (cosine, default 0.90) decides cache hits. Applied only on the non-streaming `RAGPipeline.answer()` path — SSE streaming stays uncached this week. Cache is automatically invalidated on re-ingest because `corpus_revision` is derived from `(passage_count, max_passage_id, latest_updated_at)`.

4. **LLM-as-judge verifier** — `Verifier` calls Flash-Lite with a structured JSON prompt after generation. Returns `is_faithful`, `has_sufficient_evidence`, `unsupported_claims`, and `rewrite_suggestions`. Fails open on malformed JSON — the answer is surfaced rather than silently dropped.

5. **Verifier/refusal wiring** — `RAGResponse` gained a `status` field (`"ok"` | `"insufficient_evidence"`) and an optional `RefusalPayload`. When the verifier rejects, `answer()` returns the refusal instead of the raw answer. On the `/chat` SSE path the backend buffers generation, runs the verifier, and either replays accepted text as `token` events or emits a single `refusal` event — rejected text is never streamed to the client.

6. **Entity schema registry + extractor** — `GameAdapter` now carries an optional `entity_schema` list of typed entity definitions. The ingestion pipeline extracts entities from each chunk via a Flash-Lite call and upserts them to a new `entities` table using SELECT-then-UPDATE/INSERT (no unique constraint required).

7. **Typed tool use** — `ToolDefinition`, `ToolCall`, and `ToolDispatcher` in `app/llm/tools.py`; `GeminiProvider.complete_with_tools()` using `FunctionDeclaration` and `Part.function_call`. `RAGPipeline.answer()` runs a bounded tool loop (≤ 3 iterations, configurable) when a `ToolDispatcher` is wired in. Streaming stays tool-less this week.

8. **Hades eval dataset expanded** — 48 new questions (hades-153 to hades-200) across multi-hop, ambiguous, and adversarial strata. Final: 200 questions total, meeting stratum minimums `{factual: 60, multi_hop: 60, ambiguous: 40, adversarial: 40}`.

9. **Hades II eval dataset authored** — 50 questions (`hades2-001` to `hades2-050`) covering Melinoë, Nocturnus weapons, Arcana Cards, and Hades II-specific mechanics. Stratum distribution: `{factual: 15, multi_hop: 15, ambiguous: 10, adversarial: 10}`.

10. **Ablation harness** — `app/eval/ablations.py` runs a 9-configuration matrix (baseline, +rewriter, +rerank, +verifier, +cache, +tools, full, hybrid-no-dense, hybrid-no-bm25) and emits a Markdown report. Tool-dependent configs are skipped gracefully when the active answer provider lacks `complete_with_tools`. Cache-enabled configs clear their scope before each run to prevent cross-config hits.

11. **Frontend: `InsufficientEvidenceCard`** — Yellow-border card component rendering the refusal message, unsupported claims, and inert rewrite suggestion buttons. Wired through `ChatView` (refusal SSE event), `MessageBubble` (early-return on `refusal` prop), and `fetchSessionMessages` (refusal payload preserved on session re-open).
